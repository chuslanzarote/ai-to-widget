### Entity: Product

**Purpose.** Represents a coffee or piece of brewing equipment for sale.

**Primary table.** `product`

**Fields indexed for RAG.**

| Field | Type | Included | Rationale |
|---|---|---|---|
| `id` | text (PK) | yes, as identifier | Needed to generate links and tool calls. |
| `title` | text | yes | Primary name of the product. |
| `subtitle` | text | yes | Secondary descriptor ("Washed, 250g"). |
| `description` | text (long) | yes | Full product description with tasting notes. |
| `handle` | text (slug) | yes | Used for storefront URLs. |
| `metadata` | jsonb | yes, selected keys | Contains cupping notes, origin, altitude, process. |
| `material` | text | yes | For equipment: "glass", "ceramic", "stainless". |
| `weight` | numeric | yes | In grams. |
| `created_at` | timestamp | no | Not user-facing. |
| `updated_at` | timestamp | no | Not user-facing. |
| `status` | text | no (filter) | Used as a filter (only `published` products are indexed). |
| `discountable` | boolean | no | Internal logic. |

**Related tables joined for enrichment.**

| Table | Join | Fields used |
|---|---|---|
| `product_variant` | `product_id = product.id` | `title`, `sku`, `inventory_quantity` (for stock awareness) |
| `product_option` | `product_id = product.id` | `title`, `values` (size, grind, etc.) |
| `product_category` | `product_id = product.id` via `product_category_product` join table | `name` |
| `product_collection` | `collection_id = product.collection_id` | `title`, `handle` |
| `product_tag` | via `product_tags` | `value` |

**Document assembly.** One document per `product` row (not per variant). The document concatenates: title, subtitle, description, category names, collection title, tag values, selected `metadata` keys, and a brief summary of variants ("available in 250g and 1kg"). Variants are referenced by SKU but not treated as separate entities.

**PII flags.** None. All fields are customer-facing product data.

**Filters applied during ingestion.**
- `status = 'published'`
- `deleted_at IS NULL`

**Estimated volume.** ~300 rows after filters. ~300 documents.

## Ignored tables

The following tables are present in the schema but excluded from indexing. The rationale is documented so that a future maintainer understands the decision.

### Tables containing PII

These contain personally identifiable information and are excluded under principle P1 (never leak PII into a public-facing agent).

- `customer` — emails, phone numbers, names
- `customer_address` — shipping addresses
- `customer_group` — could leak business customer segmentation
- `address` — standalone address records
- `payment` — payment method metadata, last-four digits

### Tables containing other users' orders

Order history is per-customer and retrieved via authenticated API calls at runtime, not via RAG (which would leak one customer's orders to another).

- `order`, `order_item`, `return`, `return_item`, `refund`, `swap`, `claim_order`
- `cart`, `cart_line_item` (live per-user data; accessed via API)

### Internal / operational tables

These store implementation details, not business content.

- `migration`, `staged_job`, `job`, `event`
- `batch_job`, `notification`, `notification_provider`
- `analytics_config`, `store`
- `idempotency_key`, `session`, `user` (Medusa admin users)
- `oauth`, `publishable_api_key_sales_channel`, `api_key`

### Infrastructure tables

Medusa's built-in auxiliary tables that don't represent business entities.

- `currency`, `country`, `fulfillment_provider`, `payment_provider`, `payment_session`
- `tax_rate`, `tax_provider`, `shipping_option`, `shipping_profile`, `shipping_method`
- `sales_channel`, `sales_channel_location`, `stock_location`
- `inventory_item`, `inventory_level`, `reservation_item` (used at runtime via API, not RAG)
- `price`, `price_list`, `price_set`, `money_amount` (prices queried live, never stale in RAG)
- `discount`, `gift_card`, `discount_condition` (promotional logic, handled by API)

## Builder's overrides

*(The Builder may add notes below that Opus could not infer. These are respected on re-runs of `/atw.schema`.)*

- The `metadata` field on `product` has a key called `cupping_notes` that contains a comma-separated list of flavor descriptors. This key is especially important for flavor-profile recommendations and must be prominent in the enrichment output.
- Products with `metadata.limited_edition = true` should surface that fact in their enrichment text, because shoppers often ask about "special" or "rare" coffees.
- The `product_tag` values include internal ops tags like `needs-photo` that are not customer-facing. Only tags matching `^(origin|process|roast|method)-` should be included in enrichment.
