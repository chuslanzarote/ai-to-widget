## Data layer

### Postgres container

- Image: `pgvector/pgvector:pg16`
- Database: `agentforge`
- Exposed port: `5433` (not 5432, to avoid conflicts with the Builder's Medusa demo Postgres)
- Initialized with migrations from `packages/backend/migrations/`

### Ingested reference tables

The Builder's `pg_dump` output is imported into an internal schema called `client_ref`. Only the tables listed as "primary" or "related" in `schema-map.md` are imported. The rest are excluded at import time to keep the database small and to avoid tempting future maintainers to rely on data that wasn't reviewed.

Tables imported for the Aurelia demo:
- `client_ref.product` (~300 rows)
- `client_ref.product_variant` (~650 rows — variants of the 300 products)
- `client_ref.product_option` (~600 rows)
- `client_ref.product_category` (~25 rows)
- `client_ref.product_collection` (~12 rows)
- `client_ref.product_tag` (~80 rows, filtered to the 4 allowed tag prefixes)
- `client_ref.region` (~5 rows)
- Join tables: `client_ref.product_category_product`, `client_ref.product_tags`, etc.

### Vector index

Schema of the main indexed table:

```sql
CREATE TABLE agentforge_documents (
  id           text PRIMARY KEY,
  entity_type  text NOT NULL,       -- 'product' | 'category' | 'collection' | 'region'
  entity_id    text NOT NULL,
  document     text NOT NULL,       -- the enriched semantic text
  facts        jsonb NOT NULL,      -- verbatim facts with source field citations
  categories   jsonb NOT NULL,      -- interpretive categories (flavor profile, brewing method, etc.)
  metadata     jsonb NOT NULL,      -- any other metadata
  embedding    vector(384) NOT NULL,
  created_at   timestamp DEFAULT now(),
  updated_at   timestamp DEFAULT now()
);

CREATE INDEX agentforge_documents_embedding_idx
  ON agentforge_documents
  USING hnsw (embedding vector_cosine_ops);

CREATE INDEX agentforge_documents_entity_idx
  ON agentforge_documents (entity_type, entity_id);
```

## Embedding model

- **Model.** `Xenova/bge-small-multilingual-v1.5` via `@xenova/transformers`.
- **Dimension.** 384.
- **Rationale.** Multilingual because the Aurelia shop operates in Spanish primarily, with English fallback. The `small` variant fits in ~100MB RAM and embeds in a few milliseconds per document on CPU. Quality is sufficient for a catalog of this size; if retrieval quality is poor in verify, upgrade to `bge-base-multilingual`.
- **Where it runs.** Inside the Ai to Widget backend container. ONNX runtime. No Python, no separate service.
- **Warm-up.** The model is downloaded and cached at container build time. Startup loads it into memory (≤10s).

## Enrichment plan

### Pipeline per entity

For each entity in `schema-map.md`:

1. **Assemble.** A deterministic script reads the entity's primary table row, joins with the related tables specified in the schema map, and produces a structured input: a JSON blob with the verbatim field values.
2. **Enrich.** An Opus 4.7 call receives the structured input plus an anchored prompt (see below) and produces an enriched document plus facts plus categories.
3. **Validate.** A deterministic script checks that every fact in the output traces to a field in the input (provenance check). If any fact lacks citation, the entity is flagged for Builder review before indexing.
4. **Embed.** The enriched document is passed through the embedding model.
5. **Insert.** Upsert into `agentforge_documents` keyed by `entity_id`.

### Enrichment prompt template (per entity type)

```
You are analyzing a piece of business data for Aurelia, a boutique specialty
coffee shop. Your job is to produce an enriched semantic representation that
will be used to help shoppers find what they need.

Hard rules:
- You MUST NOT state any fact that is not directly supported by the fields below.
- Every claim in the "facts" section MUST cite which field it comes from.
- The "categories" section may assign interpretive labels (flavor families,
  brewing method suitability, use cases) but only based on evidence in the fields.
- If a field is empty or missing, do not speculate about its content.
- If you cannot produce a useful enrichment from the data (insufficient content),
  return {"insufficient_data": true} and stop.

Output format (JSON):
{
  "document": "<a 2-4 sentence natural-language description for embedding>",
  "facts": [
    { "claim": "<verbatim fact>", "source": "<field name>" },
    ...
  ],
  "categories": {
    "flavor_family": ["chocolate", "nutty", ...],
    "brewing_methods_suited": ["v60", "aeropress", ...],
    "use_case": ["daily-driver", "gift", "espresso", ...]
  }
}

Input fields:
<fields serialized as JSON>
```

Category vocabularies are fixed per entity type (not freeform) to prevent Opus from inventing categories. For products in Aurelia, the vocabularies are:

- `flavor_family`: chocolate, nutty, fruit-red, fruit-tropical, fruit-citrus, floral, spice, earthy, sweet-caramel
- `brewing_methods_suited`: espresso, moka, v60, chemex, aeropress, french-press, cold-brew
- `use_case`: daily-driver, gift, espresso-only, specialty-tasting, limited-edition

Tables of allowed categories are referenced from the prompt; deviations are rejected during validation.

### Enrichment execution

- Concurrency: 10 parallel Opus calls. Anthropic rate limits are well above this level for Tier 1+ accounts.
- Retry policy: 3 retries with exponential backoff on transient errors (429, 5xx). Fatal errors (400, auth) halt the build and report.
- Progress streaming: each completed entity writes a progress line to the Claude Code session so the Builder sees "Enriched 47/300 products (est. 15min remaining)".
- Resumability: on any restart of `/atw.build`, already-enriched entities are skipped (detected by presence of `entity_id` in `agentforge_documents`).

## Backend generation

The Ai to Widget runtime backend is a Fastify TypeScript service. `/atw.build` generates it from templates:

- `packages/backend/src/server.ts.hbs` → `backend/src/server.ts`
- `packages/backend/src/tools.ts.hbs` → `backend/src/tools.ts` (rendered with the tools from `action-manifest.md`)
- `packages/backend/src/prompts.ts.hbs` → `backend/src/prompts.ts` (rendered with the system prompt from `action-manifest.md`)
- `packages/backend/src/config.ts.hbs` → `backend/src/config.ts` (rendered with Postgres connection info, API keys, etc.)

Generated code is placed under `backend/` in the Builder's project. It is **owned** by the Builder after generation — re-runs of `/atw.build` re-render only if the source templates or inputs have changed, with a clear prompt before overwriting.

### Backend image

A local Dockerfile builds the Node service. Base: `node:20-alpine`. The image includes the compiled service plus the cached embedding model.

## Widget generation

The widget is a single-file JS bundle compiled by `esbuild`.

- Entry: `packages/widget/src/index.ts`
- Output: `dist/widget.js` (IIFE format, self-contained)
- Output size target: ≤ 80KB gzipped
- No framework dependencies (no React, no Vue). UI uses vanilla DOM + a small set of helper functions.

Template values injected at build:
- Backend URL (from `build-plan.md`, default `http://localhost:3100`)
- Agent display name and branding ("Aurelia Brew Guide")
- CSS custom properties for theme customization
- Feature flags (confirmation UI enabled, suggestions enabled, etc.)

The widget also ships with `dist/widget.css` containing the default theme. The host can override any of the CSS custom properties.

## Docker Compose orchestration

After build, the Builder has a `docker-compose.yml` at the project root that launches:

```yaml
services:
  agentforge_postgres:
    image: pgvector/pgvector:pg16
    ports: ["5433:5432"]
    environment:
      POSTGRES_DB: agentforge
      POSTGRES_USER: agentforge
      POSTGRES_PASSWORD: agentforge_local
    volumes:
      - agentforge_pg_data:/var/lib/postgresql/data
      - ./backend/migrations:/docker-entrypoint-initdb.d

  agentforge_backend:
    build: ./backend
    ports: ["3100:3100"]
    environment:
      DATABASE_URL: postgres://atw:agentforge_local@agentforge_postgres:5432/atw
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      ALLOWED_ORIGINS: http://localhost:9000,http://localhost:8000
    depends_on:
      - agentforge_postgres

volumes:
  agentforge_pg_data:
```

The widget (`dist/widget.js`) is a static file served by whatever HTTP server the Builder wishes. In the Medusa demo, it is copied to the Next.js storefront's `public/` directory.

## Build sequence

Execution order for `/atw.build`:

1. Validate inputs (all required artifacts present).
2. Start Postgres container (if not running).
3. Apply migrations (creates `agentforge_documents` table and indexes).
4. Import reference tables from SQL dumps into `client_ref` schema.
5. Load embedding model (downloads once, caches thereafter).
6. Enrich entities (concurrent, resumable, validated).
7. Compute embeddings and upsert documents.
8. Render backend templates with values from artifacts.
9. Render widget bundle with configuration.
10. Build backend Docker image.
11. Write `.atw/state/build-manifest.json` (what was built, when, with what inputs).
12. Report to the Builder: summary of what was built, where it lives, how to run it.

Estimated total duration for the Aurelia demo (~342 entities): **12-18 minutes** on a developer laptop. Most of the time is enrichment (Opus calls dominate); embedding and insertion are sub-second per entity.

## Failure modes and handling

| Failure | Response |
|---|---|
| Anthropic API auth error | Halt immediately; clear message to Builder. |
| Anthropic rate limit | Back off and retry up to 3 times. |
| Validation failure for a specific entity (fact without citation) | Log, skip entity, continue with others. Builder sees the list at the end and can decide. |
| Postgres connection error | Halt, print debugging info (is container running?). |
| Out of memory during enrichment | Lower concurrency (default 10 → 3) and retry. |
| Embedding model fails to load | Halt, print the download URL so Builder can manually fetch. |

## Idempotency behavior

On re-run of `/atw.build`:

- Artifact files are re-read; if unchanged since the last build, skip their corresponding generation steps.
- `agentforge_documents` rows with existing `entity_id` are not re-enriched unless `--force` is passed or the entity's source row has a newer `updated_at`.
- Backend template rendering is a no-op if inputs and templates are unchanged.
- Widget compilation is a no-op if no source changed.
- The state file tracks what was done in the last run to enable this logic.

## Stretch decisions (not in V1)

These are captured here so the Builder and future maintainers see what was considered:

- **Hybrid retrieval (BM25 + vector).** Adds a keyword index alongside pgvector. Deferred — current pure-vector retrieval is sufficient for the demo dataset.
- **Reranking.** Could improve top-K quality. Deferred — not on the hackathon scope.
- **Webhooks for cart/order updates.** Would keep the RAG fresher. Deferred — CDC is roadmap.
- **Multiple enrichment passes.** First pass extracts facts, second pass writes the narrative. Could improve quality but doubles cost. Deferred.
