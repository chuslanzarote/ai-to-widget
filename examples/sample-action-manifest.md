## Tools

### `search_products`

**Purpose.** Full-text and filter search over the product catalog.

**HTTP mapping.** `GET /store/products`

**Requires auth.** No.

**Requires user confirmation.** No (read-only).

**Parameters.**

```json
{
  "type": "object",
  "properties": {
    "q": {
      "type": "string",
      "description": "Free-text search query over product title and description. Use for name matching."
    },
    "category_id": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Filter by category IDs. Obtain these from list_categories."
    },
    "collection_id": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Filter by collection IDs."
    },
    "tags": {
      "type": "array",
      "items": { "type": "string" },
      "description": "Filter by tag values (e.g., 'origin-ethiopia', 'roast-light')."
    },
    "limit": {
      "type": "integer",
      "default": 10,
      "maximum": 50
    },
    "offset": {
      "type": "integer",
      "default": 0
    }
  }
}
```

**Description given to Opus.** *"Search the product catalog by text or filters. Use this when the shopper describes what they want in terms that match product titles, descriptions, or structured attributes (origin country, roast level, category). For vague or flavor-based queries, prefer `semantic_search_products` instead."*

**Notes.** This is the native Medusa search. It works well for exact matches and filter combinations, poorly for natural-language descriptions. For the latter, the RAG path is preferred; this tool is used when the agent has already narrowed down to structured criteria.

### `add_to_cart`  ⚠️ requires confirmation

**Purpose.** Add a product variant to the cart.

**HTTP mapping.** `POST /store/carts/{cart_id}/line-items`

**Requires auth.** Yes (session-based or cart cookie).

**Requires user confirmation.** **Yes.** The widget MUST display a confirmation UI with the product, variant, quantity, and price before executing.

**Parameters.**

```json
{
  "type": "object",
  "required": ["cart_id", "variant_id", "quantity"],
  "properties": {
    "cart_id": {
      "type": "string",
      "description": "Cart ID from widget context."
    },
    "variant_id": {
      "type": "string",
      "description": "The specific variant ID. Obtain from get_product, which returns variants. Do not confuse with product_id."
    },
    "quantity": {
      "type": "integer",
      "minimum": 1,
      "maximum": 20,
      "description": "Number of units to add."
    }
  }
}
```

**Description given to Opus.** *"Propose adding a product variant to the cart. The user will be shown a confirmation UI before the action executes. Always call `get_product` first to confirm the variant_id and current price. If the shopper asks for 'the 250g bag', look up which variant corresponds to that option."*

**Confirmation card.** The widget renders a card showing: product title, variant title (e.g., "250g"), quantity, unit price, total, and two buttons: "Add to cart" and "Cancel". The user MUST click "Add to cart" for the call to proceed.

**Notes.** The `cart_id` is always available in the widget context. If not present (new shopper), the widget creates a cart first via `POST /store/carts` as a transparent pre-step, without involving the agent.

## Excluded endpoints

The OpenAPI spec contains 116 endpoints that are not exposed to the agent. Grouped:

### Admin API (94 endpoints, none exposed)

All `/admin/*` endpoints are admin-only and are never exposed to a customer-facing agent. They would allow the agent, if prompt-injected, to perform destructive operations. Principle P2.

### Authentication flows (8 endpoints)

Login, logout, password reset, signup. These are driven by Medusa's own auth UI and are not delegated to an agent.

- `POST /store/auth`, `DELETE /store/auth`, `POST /store/customers/password-token`, etc.

### Payment and checkout (10 endpoints)

Payment is a regulated flow with specific UI requirements. The agent can guide a shopper toward checkout, but the checkout itself happens in the native Medusa UI.

- `POST /store/carts/{id}/payment-sessions`, `POST /store/carts/{id}/complete`, etc.

### Returns, swaps, and disputes (4 endpoints)

Too high-risk to delegate to an agent in V1. Direct users to human support.

### Analytics and internal health (4 endpoints)

`/internal/*` and `/_health` endpoints are infrastructure, not business functionality.
