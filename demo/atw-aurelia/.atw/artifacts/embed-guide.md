## 1. Copy the bundle into `public/`

```bash
cp dist/widget.js public/widget.js
cp dist/widget.css public/widget.css
```

The bundle is deterministic (see Feature 002 determinism contract), so
you can commit the copied files to git or rebuild them whenever the
Builder's `/atw.build` runs.

## 2. Load the widget from your root layout

`app/layout.tsx`:

```tsx
import Script from "next/script";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en-US">
      <body>
        {children}
        <link rel="stylesheet" href="/widget.css" />
        <Script
          src="/widget.js"
          strategy="afterInteractive"
          data-backend-url="http://localhost:8080"
          data-api-base-url="http://localhost:8000"
          data-auth-mode="cookie"
          data-launcher-position="bottom-right"
          data-locale="en-US"
          data-login-url="http://localhost:8000/account"
        />
      </body>
    </html>
  );
}
```

## 3. CORS on your host API

The widget will call your host API directly from the browser for actions.
Make sure the host returns the right headers:

- `Access-Control-Allow-Origin: <your storefront origin>`
- `Access-Control-Allow-Methods: GET, POST, PATCH, DELETE, OPTIONS`
- `Access-Control-Allow-Headers: Content-Type, Authorization`
- `Access-Control-Allow-Credentials: true` *(required for cookie mode)*

## 4. CORS on the ATW backend

Set `ALLOWED_ORIGINS` on the ATW backend to the origin(s) where your
storefront is served. Comma-separated. Example:

```env
ALLOWED_ORIGINS=https://your-storefront.example.com
```

## 5. Theming

The widget honours a small set of CSS custom properties. Override them on
your storefront's CSS to match the host theme:

```css
:root {
  /* add or override any other --atw-* variable as needed */
}
```

## 6. Troubleshooting

- **Launcher does not appear** — open DevTools → console, look for a
  `[atw]` error. The most common cause is a missing or malformed
  `data-backend-url`.
- **CORS error** — the ATW backend rejected the origin. Double-check
  `ALLOWED_ORIGINS` matches the URL in the browser address bar exactly.
- **401 on actions** — either the shopper is not signed into the host
  app (widget should show "please log in first"), or the host API
  cookie is being blocked by SameSite policy. In cookie mode, the
  storefront and host API must be on the same site or the host API
  must respond with `Access-Control-Allow-Credentials: true` and
  `Access-Control-Allow-Origin` set to a specific origin (not `*`).
- **`ATW_TOOL_NOT_ALLOWED` in the console** — the backend tried to
  emit an action for a tool outside the widget allowlist. This is a
  defence-in-depth refusal; inspect the action-manifest.md that was
  rendered into the build.
