## Data layer

### Postgres container (`atw_postgres`)

- Image: `pgvector/pgvector:pg16`
- Database: `atw`
- Exposed port: `5433` (the shop's own Postgres already uses `5434`, so there is no conflict)
- Initialised with migrations from `packages/backend/migrations/`

### Ingested reference tables

Only `products` is imported into the `client_ref` schema. `carts`, `cart_items`, `orders`, `order_items`, and `users` are explicitly excluded per `schema-map.md` — they are either PII or per-shopper runtime state. Feature 007's model is *"data the agent reads about other shoppers never enters the RAG; such data is fetched live via widget-executed tools"*.

Tables imported:
- `client_ref.products` (~20 rows from the seed)

### Vector index

```sql
CREATE TABLE atw_documents (
  id           text PRIMARY KEY,
  entity_type  text NOT NULL,       -- 'products'
  entity_id    text NOT NULL,       -- product UUID
  document     text NOT NULL,
  facts        jsonb NOT NULL,
  categories   jsonb NOT NULL,
  metadata     jsonb NOT NULL,
  embedding    vector(384) NOT NULL,
  created_at   timestamp DEFAULT now(),
  updated_at   timestamp DEFAULT now()
);

CREATE INDEX atw_documents_embedding_idx
  ON atw_documents USING hnsw (embedding vector_cosine_ops);

CREATE INDEX atw_documents_entity_idx
  ON atw_documents (entity_type, entity_id);
```

## Embedding model

- **Model.** `Xenova/bge-small-multilingual-v1.5` via `@xenova/transformers`.
- **Dimension.** 384.
- **Rationale.** The agent is monolingual English at v1 but the multilingual variant is the package's canonical default and has no downside for English. Sufficient quality for a 20-product catalogue; swap to `bge-base-multilingual` if retrieval quality is poor during verify.
- **Where it runs.** Inside the `atw_backend` container via ONNX runtime. No Python, no separate service.
- **Warm-up.** Model downloaded and cached at container build time; startup loads it into memory (≤10s).

## Enrichment plan

### Pipeline per entity

For each entity in `schema-map.md` classified `indexable`:

1. **Assemble.** Deterministic script reads the entity's primary table row and produces a structured JSON blob with the verbatim field values. `Products` has no joined references, so this is a single-row read.
2. **Enrich.** An Opus 4.7 call receives the structured input plus an anchored prompt (see below) and produces an enriched document plus facts plus categories.
3. **Validate.** Deterministic script checks that every fact cites a field in the input (provenance check). Uncited facts flag the entity for Builder review.
4. **Embed.** The enriched document is passed through the embedding model.
5. **Insert.** Upsert into `atw_documents` keyed by `entity_id`.

### Enrichment prompt template (Products)

```
You are analysing a piece of business data for a coffee ecommerce specialised
in baristas and gourmet coffee tasters. Your job is to produce an enriched
semantic representation that will be used to help shoppers find what they
need.

Hard rules:
- You MUST NOT state any fact that is not directly supported by the fields below.
- Every claim in the "facts" section MUST cite which field it comes from.
- The "categories" section may assign interpretive labels (flavor families,
  brewing method suitability, use cases) but only based on evidence in the fields.
- If a field is empty or missing, do not speculate about its content.
- If you cannot produce a useful enrichment from the data (insufficient content),
  return {"insufficient_data": true} and stop.

Output format (JSON):
{
  "document": "<a 2-4 sentence natural-language description for embedding>",
  "facts": [
    { "claim": "<verbatim fact>", "source": "<field name>" }
  ],
  "categories": {
    "flavor_family": [...],
    "brewing_methods_suited": [...],
    "use_case": [...]
  }
}

Input fields:
<serialised fields from the `products` row: name, handle, description,
 price_cents, image_url, in_stock>
```

### Category vocabularies (fixed; no invented labels)

- `flavor_family`: chocolate, nutty, fruit-red, fruit-tropical, fruit-citrus, floral, spice, earthy, sweet-caramel
- `brewing_methods_suited`: espresso, moka, v60, chemex, aeropress, french-press, cold-brew
- `use_case`: daily-driver, gift, espresso-only, specialty-tasting, limited-edition

```
Enrichment calls:     20
Per-call cost (USD):  0.0300
Subtotal (USD):       0.60
Retry buffer (USD):   0.12  (+20%)
Estimated total:      0.72
```

| Entity type | Count | Cost per call (USD) | Subtotal |
|---|---|---|---|
| Products | 20 | ~$0.03 | ~$0.60 |
| Retry buffer (+20%) | — | — | ~$0.12 |

### Enrichment execution

- Concurrency: 10 parallel Opus calls (overkill for 20 items but keeps the default).
- Retry policy: 3 retries with exponential backoff on transient errors (429, 5xx). Fatal errors (400, auth) halt the build.
- Progress streaming: per-entity progress line.
- Resumability: on restart, already-enriched entities are skipped (detected by `entity_id` in `atw_documents`).

## Docker Compose orchestration

After build, `docker-compose.yml` at the project root launches:

```yaml
services:
  atw_postgres:
    image: pgvector/pgvector:pg16
    ports: ["5433:5432"]
    environment:
      POSTGRES_DB: atw
      POSTGRES_USER: atw
      POSTGRES_PASSWORD: atw
    volumes:
      - atw_pg_data:/var/lib/postgresql/data
      - ./backend/migrations:/docker-entrypoint-initdb.d

  atw_backend:
    build: ./backend
    ports: ["3100:3100"]
    environment:
      DATABASE_URL: postgres://atw:atw@atw_postgres:5432/atw
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}
      ALLOWED_ORIGINS: http://localhost:5173,http://localhost:8080
    depends_on:
      - atw_postgres

volumes:
  atw_pg_data:
```

The shop SPA (`demo/shop/frontend`) loads `dist/widget.js` as a static asset. The widget does not have its own origin — it inherits the SPA's origin. That's why `ALLOWED_ORIGINS` on the ATW backend points at the SPA's Vite dev server (`http://localhost:5173`) and compose-mode nginx (`http://localhost:8080`). The **shop backend** (port 3200) separately manages CORS for the SPA; that is outside the scope of this plan.

## Build sequence

1. Validate inputs (all required artifacts present).
2. Start `atw_postgres` container (if not running).
3. Apply migrations (creates `atw_documents` table and indexes).
4. Import `products` from `.atw/inputs/shop-products.sql` into the `client_ref` schema.
5. Load embedding model (downloads once, caches thereafter).
6. Enrich entities (20 products, concurrent, resumable, validated).
7. Compute embeddings and upsert documents.
8. Render backend templates with values from artifacts.
9. Render widget bundle with configuration.
10. Build `atw_backend` Docker image.
11. Write `.atw/state/build-manifest.json`.
12. Report summary to the Builder.

Estimated total duration for 20 entities: **~2–4 minutes** on a developer laptop. Enrichment dominates; embedding and insertion are sub-second per entity.

## Failure modes and handling

| Failure | Response |
|---|---|
| Anthropic API auth error | Halt immediately; clear message to Builder. |
| Anthropic rate limit | Back off and retry up to 3 times. |
| Validation failure for an entity (fact without citation) | Log, skip entity, continue. Builder sees the list at the end. |
| Postgres connection error | Halt; print debugging info. |
| Embedding model fails to load | Halt; print the download URL. |

## Idempotency behaviour

On re-run of `/atw.build`:

- Artifact files are re-read; unchanged artifacts skip their corresponding generation steps.
- `atw_documents` rows with existing `entity_id` are not re-enriched unless `--force` is passed or the source row's `updated_at` is newer.
- Backend template rendering is a no-op if inputs and templates are unchanged.
- Widget compilation is a no-op if no source changed.

## Preparing the SQL dump

`/atw.build` expects `.atw/inputs/shop-products.sql` to exist before it runs. The current tooling does **not** generate this dump interactively — the Builder must produce it manually. Below are the exact steps.

### Prerequisites

1. The reference shop is running with its database seeded. From `demo/shop/`:

   ```bash
   docker compose up -d
   # wait for shop_postgres to become healthy; the seed runs during shop_backend's first boot
   ```

2. `shop_postgres` holds the 20 seeded products. Verify with:

   ```bash
   docker compose -f demo/shop/docker-compose.yml exec shop_postgres \
     psql -U shop -d shop -c "SELECT count(*) FROM products;"
   ```

### Generate the dump

From `demo/atw-shop-host/` (this directory):

**Option A — `pg_dump` installed on the host (port 5434 is exposed by the compose).**

bash / zsh:

```bash
PGPASSWORD=shop_local pg_dump \
  -h localhost -p 5434 \
  -U shop -d shop \
  --data-only --inserts \
  --table=products \
  > .atw/inputs/shop-products.sql
```

PowerShell (Windows):

```powershell
$env:PGPASSWORD = "shop_local"
pg_dump -h localhost -p 5434 -U shop -d shop --data-only --inserts --table=products `
  | Out-File -Encoding utf8NoBOM .atw/inputs/shop-products.sql
```

Do **not** use plain `>` in PowerShell — it writes UTF-16 LE and breaks the importer. `Out-File -Encoding utf8NoBOM` is required (PowerShell 7+); on PowerShell 5 use `-Encoding utf8` and manually strip the BOM with `(Get-Content …) | Set-Content -Encoding utf8`.

**Option B — run `pg_dump` inside the container (no host client needed).**

bash / zsh:

```bash
docker compose -f ../shop/docker-compose.yml exec -T shop_postgres \
  pg_dump -U shop -d shop --data-only --inserts --table=products \
  > .atw/inputs/shop-products.sql
```

PowerShell (Windows):

```powershell
docker compose -f ..\shop\docker-compose.yml exec -T shop_postgres pg_dump -U shop -d shop --data-only --inserts --table=products `
  | Out-File -Encoding utf8NoBOM .atw/inputs/shop-products.sql
```

### Verify

bash / zsh:

```bash
grep -c "^INSERT INTO" .atw/inputs/shop-products.sql
# expected output: 20
```

PowerShell:

```powershell
(Select-String -Pattern "^INSERT INTO" -Path .atw/inputs/shop-products.sql).Count
# expected output: 20
```

### Flags explained

- `--data-only`: the schema is already described in `schema-map.md`; the build pipeline creates `client_ref.products` from its own migration and only needs the rows.
- `--inserts`: use `INSERT` statements (not `COPY`) so the import is tolerant of schema-name differences between the source and target.
- `--table=products`: limit the dump to the one indexable table. `users`, `carts`, `cart_items`, `orders`, `order_items` are intentionally excluded — they are PII (`users`) or per-shopper runtime state (the rest) per `schema-map.md`.

If you re-seed the shop (e.g. `docker compose down -v && docker compose up -d`), re-run the dump. `/atw.build` detects changes via `source_hash` and only re-enriches affected rows on the next run.
