# Contract: `.atw/artifacts/host-requirements.md`

**Feature**: 008-atw-hardening
**Status**: New artefact introduced in this feature (FR-003).
**Producer**: `packages/scripts/src/atw-api.ts` (via `/atw.api`).
**Consumers**: Builder (read directly); Claude (read during post-setup diagnosis).

## Emission conditions

The file is emitted when, and only when, `deploymentType === "customer-facing-widget"` is set in `.atw/config/project.md`. On every `/atw.api` run the file is regenerated in full; Builder edits are not preserved.

## File structure

```markdown
# Host Requirements

<!-- Generated by /atw.api — edits here are overwritten on next run -->

## CORS origins

The host MUST serve responses with `Access-Control-Allow-Origin` matching one of:
- `<storefrontOrigins[0]>`
- `<storefrontOrigins[n]>`

## Preflight

The host MUST permit these verbs on preflight (`OPTIONS`):
- `GET`, `POST`, `PATCH`, `DELETE`, `OPTIONS`

The host MUST permit these headers on preflight:
- `Authorization`
- `Content-Type`
- `<any additional header discovered from the OpenAPI document>`

## Bearer-token storage

The widget reads the shopper's bearer token from `window.localStorage["<authTokenKey>"]`.
The storefront SPA MUST write the token to this key on successful login.

## Login redirect

On 401 from any authed tool call, the widget redirects to:
- `<loginUrl>` (or: unset — widget shows inline "please log in" without redirect)

## Tool-specific prerequisites

<omitted if no catalog entry declares a hostPrerequisite; otherwise one bullet per entry>
- `<toolName>`: `<hostPrerequisite text>`
```

## In-flow Builder summary

At emission time, `/atw.api` prints a short in-terminal summary:

```
Your host must provide:
  • CORS for <origin list>
  • localStorage key "<authTokenKey>" populated on login
  • login redirect to <loginUrl or "—">

Full checklist: .atw/artifacts/host-requirements.md
```

## Post-setup diagnostic use

When the Builder asks Claude "why doesn't my widget work?", Claude's first step is to read `host-requirements.md` and compare each item against the real host:

1. Open `host-requirements.md`.
2. For each listed requirement, verify (via browser devtools network tab or direct inspection) that the host satisfies it.
3. Report the first unmet requirement as the likely root cause.

This replaces the Feature 007 diagnosis pattern of re-deriving requirements from spec prose each time.

## Contract tests

1. **Emission gate.** File is NOT emitted when `deploymentType` is absent from `project.md`.
2. **Emission content.** File is emitted with every section populated from `project.md` + action catalog.
3. **Regeneration.** Running `/atw.api` twice against unchanged inputs produces byte-identical files.
4. **Override warning.** File contains the `<!-- Generated by /atw.api — edits here are overwritten on next run -->` comment.
