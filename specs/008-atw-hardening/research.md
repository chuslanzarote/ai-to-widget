# Phase 0 Research: ATW Hardening

**Feature**: 008-atw-hardening
**Date**: 2026-04-24

## Purpose

Record the non-obvious decisions behind each hardening fix. The spec carries five resolved clarifications; this document adds the remaining design-level decisions that fell out of the Technical Context and the Constitution Check, so a later reviewer can reconstruct the reasoning without re-reading the Feature 007 demo trace. Every entry follows **Decision / Rationale / Alternatives considered**. No `NEEDS CLARIFICATION` markers remain.

---

## R1. Where the `deploymentType` flag lives and what it gates

**Decision**: A single project-level string field `deploymentType` in the YAML frontmatter of `.atw/config/project.md`. Working v1 value: `customer-facing-widget`. Captured interactively during `/atw.init` (default: `customer-facing-widget` — the feature's primary target persona). When this field is `customer-facing-widget`:

- The classifier's Stage 1 rule 2 (bearer-JWT rejection, `non-cookie-security`) is **not** applied — shopper-scoped operations declaring `bearerAuth` are accepted into the widget tool catalog (FR-010).
- `/atw.api` emits `host-requirements.md` (FR-003).
- The embed template emits `data-auth-token-key` (FR-014) and `data-allowed-tools` (FR-015).

When `deploymentType` is absent or set to any other value, the classifier's original bearer-JWT rejection rule applies unchanged — protecting server-to-server or admin-console deployments where widget-side bearer handling is not the intent.

**Rationale**:
- **Clarification Q1.** The user explicitly chose a single project-level flag over per-operation gating ("per-operation flag can be layered later without schema change").
- **Principle II.** One field in one markdown file, visible and editable in a diff. No hidden environment variable, no config-table schema.
- **Principle III (Idempotent and Interruptible).** A scalar flag in frontmatter is trivially re-readable on `/atw.init` re-run; FR-005a pre-fills it. No migration needed for projects that haven't adopted it — its absence reproduces the pre-008 classifier behaviour.
- **Principle VIII.** Deterministic input → deterministic output. Given the flag's value, the classifier, API stage, and embed template all produce byte-identical artefacts.

**Alternatives considered**:
- **Per-operation `shopperBearerAllowed: true` annotation.** Rejected: noisier in the action manifest; the Builder's intent ("this project builds a customer-facing widget") is genuinely project-scoped.
- **Environment variable.** Rejected: violates Principle II (hidden state not in markdown).
- **Inferred from presence of a `bearerAuth` security scheme in OpenAPI.** Rejected: false positives for admin-console deployments that also use bearer auth but should not expose shopper tables to a widget.

## R2. `host-requirements.md` shape

**Decision**: Markdown artefact emitted by `/atw.api` when `deploymentType: customer-facing-widget` is set. Location: `.atw/artifacts/host-requirements.md`. Structure:

```markdown
# Host Requirements

<!-- Generated by /atw.api — edits here are overwritten on next run -->

## CORS origins
- `<storefrontOrigins from project.md>`

## Preflight
- Allowed headers: `Authorization`, `Content-Type`
- Allowed verbs: `GET`, `POST`, `PATCH`, `DELETE`, `OPTIONS`

## Bearer-token storage
- `localStorage` key: `<authTokenKey — default shop_auth_token>`
- Header format: `Authorization: Bearer <token>`

## Login redirect
- URL: `<loginUrl from project.md>`

## Tool-specific prerequisites
- `<tool name>`: `<one-line prerequisite>` (one line per entry derived from the action catalog's `hostPrerequisite` field, when set)
```

On each `/atw.api` run the file is regenerated in full. An inline one-line summary ("Your host must provide: CORS for <origin>, localStorage key `<key>`, login URL `<url>` — see `.atw/artifacts/host-requirements.md` for full checklist") is printed to the Builder at emission time (FR-003 in-flow summary requirement).

**Rationale**:
- **Principle II.** Human-readable markdown.
- **Principle IV.** The file exists so Claude can check it against the real host during post-setup diagnosis ("first thing compared" per the spec's FR-003 description) instead of re-deriving requirements each time.
- **Auditable.** A Builder reviewing a PR sees a diff on `host-requirements.md` when the shop's OpenAPI changes its preflight contract.
- **Deterministic emission.** Generated from `project.md` + OpenAPI + action catalog; reproducible.

**Alternatives considered**:
- **JSON artefact.** Rejected: Principle II — structured markdown is the project convention.
- **Inline in embed-guide only.** Rejected: duplicates data into two generated files that drift. The embed-guide references this artefact.
- **Builder-authored only.** Rejected: the point is that the tool derives host contracts automatically, not the Builder copying them from docs.

## R3. `crossValidateAgainstOpenAPI` credential-source backfill

**Decision**: Extend `crossValidateAgainstOpenAPI` in `packages/scripts/src/parse-action-manifest.ts` so it copies each operation's `security` list from the loaded OpenAPI document onto the corresponding catalog entry (`entry.source.security`). The existing `render-executors.ts` then emits a populated `credentialSource` block for every entry whose `security` list is non-empty — closing the B8 gap (#15) where `render-executors.ts` would silently omit the block for any entry whose `source.security` was not hand-set.

When `crossValidateAgainstOpenAPI` detects one or more operations that declare bearer security in the OpenAPI document but which (a) have no catalog entry, or (b) have a catalog entry whose `source.security` remains empty after backfill (e.g., the OpenAPI doesn't declare security on that op but the operation is shopper-scoped), the stage halts with:

```
ERROR: The following tool(s) would ship without a credential source:
  - <toolName> (<HTTP verb> <path>)
  - ...
These operations need to declare bearer security in the OpenAPI doc. Add either:
  (a) an operation-level `security: [{ bearerAuth: [] }]`, or
  (b) a global `security: [{ bearerAuth: [] }]` at the document root.

See: .atw/artifacts/host-requirements.md
```

**Rationale**:
- **Clarification Q3.** "Halt with actionable diagnostic" was the explicitly chosen path. Warning-with-fallback was ruled out.
- **Principle V.** The widget must not ship an authed tool without the declared auth source — the credential handling is anchored to what the OpenAPI document actually says.
- **Principle IV.** The diagnostic tells the Builder exactly what to change and where.
- **Mechanical enforcement.** `render-executors.ts` no longer needs to silently fall back; the invariant is enforced one step earlier.

**Alternatives considered**:
- **Warning-with-fallback.** Explicitly rejected by Clarification Q3.
- **Default `credentialSource` to bearer-localStorage for every entry when `deploymentType: customer-facing-widget`.** Rejected: violates Anchored Generation — credentials attached to operations that do not declare bearer auth would be fabricated.
- **Emit `credentialSource: null` for un-backfillable entries and let the widget surface a runtime error.** Rejected: the failure surfaces far from the root cause; catching it at build time is the safer discipline.

## R4. Chat endpoint v3 request-shape amendment

**Decision**: Extend the `ToolResultPayload` schema with two additional fields — `tool_name: string` and `tool_input: object` — carried alongside the existing `tool_use_id`, `content`, `is_error`, `status`, and `truncated`. The backend uses these to reconstruct the Anthropic message trio on receipt:

```ts
const messagesForAnthropic = [
  { role: "user", content: widgetSuppliedUserMessage },
  {
    role: "assistant",
    content: [
      {
        type: "tool_use",
        id: toolResult.tool_use_id,
        name: toolResult.tool_name,
        input: toolResult.tool_input,
      },
    ],
  },
  {
    role: "user",
    content: [
      {
        type: "tool_result",
        tool_use_id: toolResult.tool_use_id,
        content: toolResult.content,
        is_error: toolResult.is_error,
      },
    ],
  },
];
```

The backend then invokes `anthropic.messages.create({ messages: messagesForAnthropic, tools, ... })`. The widget remains unaware of Anthropic message shape — it sends `tool_name` + `tool_input` as a flat payload and receives a streamed text/citations reply (or another `action_intent` for chained tool calls).

**Rationale**:
- **Approach A (chosen in FEATURE-008 D1).** Preserves the model's voice on post-action replies; rejects the templated-reply alternative.
- **Stateless backend.** FR-018/019 requirement — the backend holds no session state; the widget supplies enough data in every POST to reconstruct the provider's view.
- **Minimal ripple.** `ConversationTurn.content` stays `string`. No widget-side type-system migration; no render-template changes.
- **Anchored to what the widget actually executed.** `tool_input` is the arguments the widget fetched with (not what Opus proposed in its original `tool_use`), so retries or user-driven confirmation edits are correctly reflected.
- **Contract alignment.** The documented request shape is rewritten to match what the code sends (FR-018) — the opposite direction of Feature 007's documented-but-never-sent typed-assistant-turn.

**Alternatives considered**:
- **Carry the full assistant-role turn with a typed `tool_use` block through `ConversationTurn`.** Rejected: would require migrating `content: string` to `content: string | ContentBlock[]`, touching every render path, loader, and prompt-composer. Explicitly out of scope per FEATURE-008 Non-goals and spec Assumptions.
- **Server-side session state keyed by `pending_turn_id`.** Rejected: violates FR-018 (stateless across posts) and Principle VII (no new session store).
- **Backend templates the post-action reply.** Rejected: FR-020 explicitly forbids templated replies.

## R5. Response-generation-failed-but-action-succeeded handshake (FR-020a)

**Decision**: When the re-invoked `anthropic.messages.create()` call fails (any non-200, network error, timeout, rate-limit) *after* the widget has already successfully completed a write-class fetch against the host, the backend:

1. Retries the call up to **2 additional times** with exponential backoff (initial 500 ms, factor 2, cap 4 s). Clarification Q4 specifies "1–2 times with backoff" — this decision lands on exactly 2 extra retries for a total of 3 attempts.
2. On continued failure, returns an HTTP 200 response with a dedicated shape:
   ```jsonc
   {
     "response_generation_failed": true,
     "action_succeeded": true,
     "pending_turn_id": null
   }
   ```
3. The widget, on seeing `response_generation_failed: true && action_succeeded: true`, renders the pinned confirmation-fallback string:
   > Action completed successfully. (Response generation failed — please refresh.)
4. The widget MUST NOT render the standard error toast or the "something went wrong on our side" template in this state.

The retry logic applies only on the post-tool-result second Opus call. The initial (pre-tool_use) Opus call retains Feature 003's existing error handling unchanged.

**Rationale**:
- **Clarification Q4.** The user explicitly chose this handshake over "widget shows generic failure".
- **Principle V.** The user-facing string is a fixed template in the widget, not an Opus-composed one — the model has already demonstrated it can't respond; fabricating prose would violate Anchored Generation. The fallback is narrow and deterministic.
- **Principle IV.** The shopper knows the action succeeded and knows to refresh — full transparency.
- **Bounded worst case.** With 500 ms → 1 s → 2 s backoff, the worst-case latency before the fallback is ≈ 3.5 s on top of the original call. Acceptable vs. incorrectly telling the shopper the action failed.

**Alternatives considered**:
- **Unlimited retry with jitter.** Rejected: unbounded worst case; the shopper is waiting.
- **Single retry.** Rejected: Clarification Q4 ("1–2 retries") permits 2; the second retry substantially improves recovery from transient provider issues.
- **Render a generic "something went wrong" and let the shopper retry the whole turn.** Rejected: implies the write failed when it did not (Clarification Q4 rationale).
- **Log the failure but don't surface it.** Rejected: the shopper needs to know the natural-language reply is missing or they will re-issue the action and potentially double-charge.

## R6. `/atw.init` re-run pre-fill policy (FR-005a)

**Decision**: On invocation, `init-project.ts` first reads any existing `.atw/config/project.md`. For every prompt it issues, if the corresponding field is populated in the existing file, it presents the value as the default in the prompt UI (`[current value] › `). Pressing Enter keeps the value. Typing a new value overwrites it. The fields covered are:

- `storefrontOrigins` (FR-001)
- `welcomeMessage` (FR-002)
- `deploymentType` (FR-010 / R1 above)
- All pre-existing `/atw.init` fields (shop name, etc.) — pre-fill behaviour extended to them for consistency.

A final diff (old vs. new) is shown before the file is written, and the Builder confirms. This is a strict extension of Principle IV (confirmation before write) over Principle III (idempotence).

**Rationale**:
- **Clarification Q5.** The user chose "in scope — pre-fill from existing values" over soft-abort or deferral.
- **Principle III.** Matches the principle's stated pattern ("Re-running `/atw.schema` loads the existing `schema-map.md`, shows what was decided, and offers refinement rather than re-asking every question").
- **Principle IV.** The final diff keeps the Builder in the loop — no silent overwrite.

**Alternatives considered**:
- **Soft abort ("project.md already exists — refusing to overwrite").** Rejected by Clarification Q5.
- **Full re-ask every field.** Rejected by Clarification Q5; also frustrating UX.
- **Flag-gated re-run (`--reset` forces re-ask).** Possible follow-up; not required for this feature.

## R7. Schema-map heading convention (FR-009)

**Decision**: Standardise on **H2** (`## Entity: <name>`) as the parser convention. Update `examples/sample-schema-map.md` and any generated sample output to match. Keep the parser at `extractSections(tree, { depth: 2 })`. Add a post-parse assertion: if the parsed entity count is zero and the raw markdown text contains `### Entity:` substrings, the parser emits:

```
ERROR: Zero entities parsed from schema-map.md. Detected H3 "### Entity:" headings — the
parser expects H2 "## Entity:". Convert your headings one level up, or regenerate the file
with /atw.schema.
```

If the parsed entity count is zero and no `### Entity:` substring is found, a different diagnostic names the convention and points at the sample file.

**Rationale**:
- **Principle II.** The parser already uses H2; the sample was wrong. Changing the sample is the smaller, safer edit.
- **Principle IV.** Loud failure at the point of parse is the spec's explicit requirement (FR-009 "fail loudly").
- **Auditable migration.** A Builder re-running `/atw.schema` regenerates a compliant `schema-map.md`; hand-authored files need a trivial H3→H2 search-and-replace, guided by the diagnostic.

**Alternatives considered**:
- **Teach the parser to accept H3.** Rejected: both conventions co-existing invites future drift. One convention is the Principle VI discipline.
- **Silently coerce H3 to H2.** Rejected: FR-009 forbids silent coercion.

## R8. `runtime-only` flag on action-manifest tool groups (FR-012)

**Decision**: Action-manifest tool-group headers gain an optional inline flag: `## Tools: <group> (runtime-only)`. When present, `validate-artifacts.ts`' `action-references-excluded-entity` rule permits the group's entries to reference entities that are absent from `schema-map.md` or explicitly listed in its `excluded` section. The flag is preserved round-trip by the manifest parser and surfaces as `runtimeOnly: true` on the rendered `action-executors.json` entry group (for debuggability — it is not read by the widget).

`crossValidateAgainstOpenAPI` recognises `runtime-only` groups as a documentation signal that the entities they target legitimately live outside the indexed data (per-shopper carts, orders, etc.) — the safe-read data-plane never sees them; they exist only as widget-executed tool calls.

**Rationale**:
- **Principle II.** One declarative marker in markdown. The signal is readable by a human reviewer.
- **Bounded scope.** The flag governs only the validator's excluded-entity rule; it does not change widget behaviour.
- **Principle V adjacent.** Widget-executed tools still trace to OpenAPI operations; the flag relaxes the *indexed-schema* cross-reference rule, not the OpenAPI-traceability rule.

**Alternatives considered**:
- **Infer runtime-only from `credentialSource` presence.** Rejected: correlated but not equivalent; a runtime-only group could target public unauthed endpoints.
- **Remove the validator rule entirely.** Rejected: the rule catches genuine bugs for safe-read tool groups; gating it with an explicit flag preserves its value.
- **Use a separate manifest section.** Rejected: splits context; the header flag is co-located with the group definition.

## R9. Singular/plural name normalisation (FR-011)

**Decision**: Extend the cross-validator's name comparison to tolerate English singular/plural variants. A new helper `packages/scripts/src/lib/singular-plural.ts` exposes:

```ts
export function normaliseName(input: string): string {
  // lowercase, strip non-alphanumerics, then apply a conservative
  // English inflection: products → product, categories → category,
  // users → user, queries → query. Non-English inputs fall through
  // unchanged; compound names are normalised word-by-word.
}
```

`normaliseName("Products") === normaliseName("Product")` → `true`. The helper is invoked at every cross-validation step that compares classifier tag names to schema-map entity names.

**Rationale**:
- **Principle VI.** A small deterministic helper. No library dependency; the conservative English rules cover the observed Feature 007 mismatches (`products`/`Product`, `orders`/`Order`, `customers`/`Customer`).
- **No semver risk.** The helper only expands the set of matches, never narrows it. Builders who already use matching names see no change.
- **Testable.** The helper is a pure function.

**Alternatives considered**:
- **`pluralize` npm package.** Rejected: introduces a runtime dependency for a few dozen English inflection rules. Principle VII (ecosystem simplicity) and Principle IX (not every problem needs a package) argue against.
- **Require one convention end-to-end.** Rejected: the classifier emits tag names from OpenAPI (where plural is conventional), the schema-map uses entity names (where singular is conventional). Forcing either upstream is churn.
- **Case-insensitive exact match only.** Rejected: leaves the demonstrated Feature 007 bug unfixed.

## R10. Thinking-indicator render timing (FR-024)

**Decision**: The widget's `panel.tsx` appends a synthetic `thinking` item to its transcript inside the same React state update that schedules the `POST /v1/chat` request. The indicator is a transcript row — not a toast, not an overlay — so screen readers announce it in the conversation flow. When the first streamed delta arrives or the final response lands, the `thinking` item is removed from the transcript in the same state update that appends the model's reply.

Because the indicator is rendered in the same microtask as the POST, a sub-second reply shows the indicator briefly (SC-009's 1-second observable guarantee). For sub-16 ms replies, the indicator may visually flicker for one frame; this is accepted rather than threshold-gated. Clarification Q2 explicitly chose "immediate — no delay threshold".

**Rationale**:
- **Clarification Q2.** Immediate display was the chosen answer.
- **Principle IV.** The shopper sees the system working at the moment they act. No silent gap.
- **Accessibility.** In-transcript placement is screen-reader-correct; a fixed toast would be announced out-of-band.

**Alternatives considered**:
- **Delay indicator by 1 s via `setTimeout`.** Rejected by Clarification Q2.
- **Show only after the first delta arrives.** Rejected: the indicator's purpose is to cover the pre-first-delta window.
- **Render as a toast outside the transcript.** Rejected: breaks the conversation accessibility model.

## R11. Action catalog `summaryTemplate` format (FR-026)

**Decision**: Each action-catalog entry may carry an optional `summaryTemplate: string` field. The value is a plain-text template with `{{ argument_name }}` placeholders resolved at render time against the action's `arguments` object. No logic, no conditionals, no partials. Example:

```jsonc
{
  "tool": "addToCart",
  "summaryTemplate": "Add {{ quantity }}× {{ product_name }} to your cart",
  "arguments": { "product_id": "...", "quantity": 1, "product_name": "Espresso" }
}
```

The widget's `ActionCard` uses `summaryTemplate` when present and falls back to the raw JSON render when absent or when any placeholder fails to resolve (undefined argument).

The template is authored at classification time (so Opus proposes it, the Builder reviews per Principle IV), stored in the action manifest, and rendered into `action-executors.json` verbatim. Argument names not declared in the entry's `arguments` schema are not resolvable and cause the fallback.

**Rationale**:
- **Principle V.** Bounded templates (no freehand LLM output at render time); the human-readable summary is authored once, reviewed once, and replayed deterministically.
- **Principle VI.** Pure-data field, deterministic rendering.
- **No new template engine.** `{{ name }}` substitution is four lines of code and avoids a Handlebars-class dependency in the widget bundle (Principle VII).

**Alternatives considered**:
- **Handlebars / Mustache in the widget.** Rejected: bundle-size cost; excessive expressiveness.
- **Freehand Opus summary per render.** Rejected: violates Anchored Generation (no grounded source at render time) and adds per-render latency.
- **Localisation placeholders.** Rejected: out of scope; follow-up.

## R12. `data-allowed-tools` derivation (FR-015)

**Decision**: `/atw.embed` reads `.atw/artifacts/action-executors.json`, extracts the `tool` field from every entry (one entry per tool), emits a comma-separated list as the `data-allowed-tools` attribute on the generated `<script>` tag. The list is sorted alphabetically for deterministic output. The widget's `config.ts` parses the attribute into an array and enforces it as the allow-list.

A tool missing from `action-executors.json` cannot end up in the attribute; a tool in the attribute but missing at runtime causes the visible-error rendering per FR-022.

**Rationale**:
- **Lockstep emission.** The action catalog is the single source; drift between allow-list and catalog is impossible by construction.
- **Principle VI.** Deterministic derivation at build time.
- **Auditable.** A Builder reviewing the embed snippet sees the exact tools permitted.

**Alternatives considered**:
- **`data-allowed-tools="*"` wildcard.** Rejected: defeats the allow-list's purpose.
- **Runtime fetch-and-parse of the catalog by the widget to build the allow-list.** Rejected: the allow-list exists specifically to protect against a compromised or missing catalog; sourcing the allow-list from the same catalog would nullify the defence.
- **Per-tool attributes (`data-allow-addToCart="true"`, etc.).** Rejected: unwieldy for ≈12 tools.

## R13. Permanent CORS wiring in `demo/shop` (FR-021)

**Decision**: Land the in-session `@fastify/cors` registration into `demo/shop/backend` permanently. `package.json` adds `"@fastify/cors": "^9.0.1"`. `src/index.ts` registers the plugin before any route with:

```ts
await app.register(cors, {
  origin: (process.env.ALLOWED_ORIGINS ?? "http://localhost:5173").split(","),
  allowedHeaders: ["Authorization", "Content-Type"],
  methods: ["GET", "POST", "PATCH", "DELETE", "OPTIONS"],
});
```

The default origin `http://localhost:5173` is the documented storefront origin; it matches the default the hardened `/atw.init` pre-fills. The env override lets operators change it without code edits.

**Rationale**:
- **Principle VIII.** A clean clone of `demo/shop` with `docker compose up` serves correct CORS headers on first boot — no in-session patch. SC-007 is a direct measurement.
- **Feature 007 architecture.** The widget executes shop API calls in the browser; CORS is non-optional.
- **Minimal surface.** One dependency, one plugin, one config block.

**Alternatives considered**:
- **Nginx / reverse-proxy CORS injection.** Rejected: adds an infrastructure layer in front of the shop; not a shop-specific implementation detail any more. Feature 008 scope is explicitly to land the fix *in the shop*.
- **Per-route CORS decorators.** Rejected: plugin-level registration covers every route uniformly.

## R14. Hash-index schema shape alignment (FR-006)

**Decision**: Standardise on `{ schema_version: "1", files: Record<relativePath, sha256Hex> }` — the shape `atw-api.ts` already writes. Update `atw-hash-inputs`'s Zod validator to match. Rename any internal references to the old `{ version: 1, entries: [...] }` shape. Add a regression contract test that parses the writer's output with the reader's validator.

**Rationale**:
- **Fewer places to edit.** `atw-api.ts` is the writer; one update to the validator (the reader) aligns both.
- **Record-shaped files-map is ergonomic.** `O(1)` lookup by path, natural JSON shape, reproducibility-friendly.
- **Zero migration.** Deployed projects never had the old `{entries: []}` reader succeed — it has been failing. Adopting the writer's shape is a net fix, not a migration.

**Alternatives considered**:
- **Keep `{version, entries}` and migrate the writer.** Rejected: more files to touch; the writer's shape is the one that exists in real `.atw/artifacts/hash-index.json` files.
- **Support both shapes.** Rejected: two shapes mean two code paths forever. Principle VI argues for one.

## R15. `--inputs` positional argument parsing (FR-007)

**Decision**: `packages/scripts/src/hash-inputs.ts`' argv parser accepts `--inputs a.md b.md c.md` (whitespace-separated positional args after the flag, terminated by another `--flag` or end-of-args). The skill docs' `/atw.plan` invocation remains the documented form. If the user passes `--inputs a.md,b.md,c.md` (legacy comma form), that is still accepted as a single arg and split on commas — but the documented form is whitespace.

**Rationale**:
- **Skill doc is the contract.** The docs document the ergonomic form; the CLI meets the doc.
- **Backwards compatible.** Comma form still works; no downstream breakage.
- **Principle IV.** The Builder sees no surprise — "Unexpected argument" errors disappear on the happy-path invocation.

**Alternatives considered**:
- **Force comma-separated, update the skill doc.** Rejected: less readable; a multi-file list with spaces reads naturally.
- **Require `--inputs=a.md,b.md`.** Rejected: uglier still.

## R16. Quoted YAML timestamps in `write-artifact` (FR-008)

**Decision**: `packages/scripts/src/write-artifact.ts`'s YAML frontmatter serializer quotes every ISO-8601 timestamp value (`"2026-04-24T15:42:00Z"` rather than `2026-04-24T15:42:00Z`). This is applied to all timestamp fields (`createdAt`, `updatedAt`, any Builder-supplied date). Non-timestamp string values continue to follow the serializer's existing quoting heuristics.

**Rationale**:
- **Fixes the Zod-date vs Zod-string mismatch.** Downstream schemas expect `z.string()`; the YAML parser's auto-typing of unquoted ISO timestamps to `Date` objects broke them.
- **No ripple.** Quoted ISO timestamps deserialise as strings in every YAML parser; nothing downstream reads them as Dates.
- **Idempotent.** Re-emitting an already-quoted timestamp is a no-op.

**Alternatives considered**:
- **Change downstream schemas to accept `z.date()`.** Rejected: broader blast radius; several downstream consumers already rely on string comparison.
- **Custom YAML loader that forces string typing.** Rejected: opaque; fixes the symptom not the cause.
