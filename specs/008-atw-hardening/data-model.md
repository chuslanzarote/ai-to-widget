# Phase 1 Data Model: ATW Hardening

**Feature**: 008-atw-hardening
**Date**: 2026-04-24

## Scope

This feature does not introduce new datastores or tables. Every change is a schema extension to an existing markdown artefact, generated-JSON artefact, or wire-format payload. Seven entities are amended; one entity is new (`host-requirements.md`).

Every entity below declares its **fields**, **validation rules**, and **state transitions** (where applicable).

---

## 1. Project config (`.atw/config/project.md`) — AMENDED

**Role**: Source of truth for all Builder-provided setup answers. Read by every downstream stage.

**Shape**: Markdown file with YAML frontmatter. The frontmatter carries typed fields; the body contains human-authored notes.

**Fields (added or materially changed in this feature)**:

| Field | Type | Required? | Default | Notes |
| --- | --- | --- | --- | --- |
| `storefrontOrigins` | `string[]` (serialized as YAML list) | Yes when `deploymentType: customer-facing-widget`; else optional | `["http://localhost:5173"]` | One or more origins the widget may run on. Threaded into render-backend `ALLOWED_ORIGINS`, into `host-requirements.md`, and into demo/shop's CORS default. |
| `welcomeMessage` | `string` | No | `"Hi! How can I help you today?"` | Surfaced by the widget on first render (FR-025). |
| `deploymentType` | `string` enum: `"customer-facing-widget"` \| (unset) | No (unset preserves pre-008 classifier behaviour) | `"customer-facing-widget"` when the Builder accepts the default during `/atw.init` | Single project-level flag (Clarification Q1 / R1). Unlocks bearer-JWT acceptance in the classifier, emission of `host-requirements.md`, and the widget-specific embed attributes. |
| `authTokenKey` | `string` | No | `"shop_auth_token"` | `localStorage` key the widget reads. Threaded into the embed template's `data-auth-token-key` (FR-014) and `host-requirements.md`. |
| `loginUrl` | `string` | No | `""` (empty — widget treats as "no redirect configured") | URL the widget redirects to on 401. Referenced by `host-requirements.md`. |
| `createdAt` | quoted ISO-8601 string | Yes | `/atw.init` emits at run time | Must be quoted in YAML (FR-008). |
| `updatedAt` | quoted ISO-8601 string | Yes | `/atw.init` emits at run time | Must be quoted in YAML (FR-008). |

Pre-existing fields (`shopName`, etc.) remain unchanged.

**Validation rules**:
- When `deploymentType === "customer-facing-widget"`:
  - `storefrontOrigins` MUST be non-empty.
  - `authTokenKey` MUST be a non-empty string.
- `createdAt` / `updatedAt` MUST parse as `z.string().datetime()` (hence the quoting requirement, FR-008).
- `storefrontOrigins` entries MUST parse as absolute URLs (scheme + host).

**State transitions**:
- Created on first `/atw.init` run.
- On re-run, `/atw.init` reads existing values, presents as pre-filled defaults (FR-005a / R6), and writes the confirmed result.
- `updatedAt` is bumped on every write.
- Deletion: the Builder may delete the file to force a full re-ask.

## 2. Host-requirements artefact (`.atw/artifacts/host-requirements.md`) — NEW

**Role**: Canonical checklist of every contract the host must satisfy for the generated widget to function. Both Builder-facing guidance and post-setup diagnostic reference.

**Shape**: Generated markdown. Overwritten on every `/atw.api` run when `deploymentType: customer-facing-widget` is set.

**Structure** (see [research.md R2](./research.md)):

```markdown
# Host Requirements

<!-- Generated by /atw.api — edits here are overwritten on next run -->

## CORS origins
## Preflight
## Bearer-token storage
## Login redirect
## Tool-specific prerequisites
```

**Fields (derived, not Builder-authored)**:

| Field | Source | Notes |
| --- | --- | --- |
| CORS origins | `project.md#storefrontOrigins` | Verbatim. |
| Allowed preflight headers | Static list + per-operation `X-*` headers discovered from the OpenAPI document's `parameters`/`requestBody` headers. | Minimum: `Authorization`, `Content-Type`. |
| Allowed verbs | Distinct HTTP verbs across all catalog entries. | Always includes `OPTIONS`. |
| Bearer-token storage key | `project.md#authTokenKey` | Verbatim. |
| Login redirect URL | `project.md#loginUrl` | Empty `—` rendering when unset. |
| Tool-specific prerequisites | Per-entry `hostPrerequisite` optional field in action-catalog entries. | Omitted entirely when no entry carries the field. |

**Validation rules**:
- File MUST be emitted only when `deploymentType: customer-facing-widget` is set.
- Overwriting is authoritative — Builder edits are not preserved across `/atw.api` runs. A header comment warns the Builder.

**State transitions**:
- Emitted / re-emitted by `/atw.api`.
- Read by Claude during post-setup diagnosis (FR-004 conversational path).
- No deletion path — `/atw.api` will re-create.

## 3. Action-executors catalog (`.atw/artifacts/action-executors.json`) — AMENDED

**Role**: Declarative catalog consumed by the widget's action-executor engine. Drives tool-call execution, allow-list derivation (`data-allowed-tools`, FR-015), and pending-action UI (summary template, FR-026).

**Shape**: JSON array of catalog entries.

**Fields (added or materially changed in this feature)**:

| Field | Type | Required? | Notes |
| --- | --- | --- | --- |
| `credentialSource` | object \| omitted | Required when the underlying OpenAPI operation declares bearer security | Shape: `{ type: "bearer-localstorage", key: string, header: "Authorization", scheme: "Bearer" }`. Populated by `crossValidateAgainstOpenAPI` backfill (R3). |
| `summaryTemplate` | string \| omitted | Optional | Plain-text template with `{{ argument_name }}` placeholders (R11). |
| `runtimeOnly` | boolean | Optional | Round-tripped from the action-manifest's `(runtime-only)` group flag (R8). Not consumed by the widget; surfaced for debuggability. |
| `hostPrerequisite` | string \| omitted | Optional | Populated by the classifier when an operation carries a documented host prerequisite. Threaded into `host-requirements.md`. |

Existing fields (`tool`, `endpoint`, `arguments`, `confirmationRequired`, `source`) remain unchanged.

**Validation rules**:
- If `entry.source.security` is non-empty, `entry.credentialSource` MUST be present with a well-formed object (FR-013). The build halts otherwise with the R3 diagnostic.
- `summaryTemplate`, if present, MUST reference only argument names declared in the entry's `arguments` schema. Unknown placeholders cause a silent fallback at render time (the widget shows raw JSON).
- `runtimeOnly: true` entries MUST trace to an OpenAPI operation; the flag does not exempt them from OpenAPI-traceability (Principle V).

**State transitions**:
- Regenerated by `/atw.build` from the action manifest + OpenAPI.
- Copied verbatim into the host's public assets (FR-016).

## 4. Action manifest (`.atw/artifacts/action-manifest.md`) — AMENDED

**Role**: Builder-facing markdown description of tool groups. Edited by hand when the Builder refines the classifier output.

**Shape**: Markdown with H2 group headers and per-tool entries.

**Fields (added in this feature)**:

- Inline header flag: `## Tools: <group> (runtime-only)` (R8). Optional; default omitted.

**Validation rules**:
- A group flagged `(runtime-only)` is permitted to reference entities absent from `schema-map.md` (FR-012).
- All existing validation rules (OpenAPI traceability, argument declaration) continue to apply.

**State transitions**:
- Written by `/atw.classify`.
- Editable by the Builder (Principle IV).
- Consumed by `/atw.build` on next run.

## 5. Chat-endpoint tool-result payload (wire format) — AMENDED

**Role**: Widget → backend payload carrying tool-call outcomes. Enables the backend to close Opus's tool-use loop without session state.

**Shape**: JSON object nested under the `tool_result` field of `POST /v1/chat` request bodies.

**Fields (added in this feature)**:

| Field | Type | Required? | Notes |
| --- | --- | --- | --- |
| `tool_name` | string | Yes on any `tool_result`-bearing POST | OperationId of the tool the widget executed. Used by the backend to reconstruct the Anthropic `tool_use` block (R4). |
| `tool_input` | object | Yes on any `tool_result`-bearing POST | The arguments the widget actually executed against the host. Used by the backend to reconstruct the Anthropic `tool_use` block (R4). Not necessarily equal to the arguments Opus originally proposed if the shopper edited them via the confirmation card. |

Pre-existing fields (`tool_use_id`, `content`, `is_error`, `status`, `truncated`) remain unchanged.

**Validation rules**:
- Backend rejects any `tool_result` missing `tool_name` or `tool_input` with HTTP 400 and a descriptive body (FR-018 contract-match).
- `tool_name` MUST match an operationId the backend recognises (derived from the shop's OpenAPI via the manifest); unknown names are 400-rejected.

**State transitions**:
- Created by the widget on each tool-call resolution (success or failure).
- Consumed by the backend in the chat-endpoint handler; discarded after the Anthropic message sequence is assembled.

## 6. Chat-endpoint response — AMENDED

**Role**: Backend → widget reply. Carries either a natural-language turn, an `action_intent`, or the new response-generation-failed fallback state (FR-020a).

**Fields (added in this feature)**:

| Field | Type | Required? | Notes |
| --- | --- | --- | --- |
| `response_generation_failed` | boolean | Present only when the FR-020a state triggers | When `true`, the widget renders the pinned confirmation-fallback string (R5). |
| `action_succeeded` | boolean | Present only alongside `response_generation_failed` | Always `true` in this state — the FR-020a handshake only fires when the underlying host write succeeded. |

Pre-existing response shapes (`text` + `citations` + `pending_turn_id`, or `action_intent` + `pending_turn_id` + `tool_call_budget_remaining`) remain unchanged.

**Validation rules**:
- `response_generation_failed: true` MUST NOT appear alongside `text` or `action_intent` — the three response shapes are mutually exclusive (widget switches on the discriminator).
- When `response_generation_failed: true`, `pending_turn_id` is always `null` (the turn is terminally over).

**State transitions**:
- Emitted only after exhausting the R5 retry budget (initial + 2 retries = 3 attempts) on the re-invoked model call.
- On receipt, the widget renders the fallback string and clears `pending_turn_id`.

## 7. Schema-map artefact (`.atw/artifacts/schema-map.md`) — AMENDED (parser only)

**Role**: Declarative record of the tables/entities the safe-read data plane indexes.

**Shape**: Markdown with H2 entity headers: `## Entity: <name>`.

**Change in this feature**:
- Parser requires H2; H3 sample is migrated to H2 (FR-009 / R7).
- Zero-entity parse emits a loud diagnostic naming the convention.

**Validation rules**:
- Parser MUST succeed with ≥ 1 entity OR fail loudly — never silently return zero.

## 8. Hash-index artefact (`.atw/artifacts/hash-index.json`) — AMENDED (reader only)

**Role**: `atw-hash-inputs` records sha256 hashes of tracked inputs.

**Shape**: JSON object.

**Fields**:

```jsonc
{
  "schema_version": "1",
  "files": {
    "<relative path>": "<sha256 hex>"
  }
}
```

**Change in this feature**:
- The `atw-hash-inputs` validator was expecting the legacy `{version, entries[]}` shape; this feature aligns it to the writer's shape (R14). The artefact structure itself is not changed.

**Validation rules**:
- Both writer (`atw-api.ts`) and reader (`hash-inputs.ts`) MUST share the same Zod schema. A contract test enforces the round-trip.

## Cross-entity invariants

- **Allow-list lockstep.** The set of tool names in `action-executors.json` MUST equal the set emitted in the embed snippet's `data-allowed-tools` attribute (R12 / FR-015). Drift at build time is impossible by construction — both are emitted from the catalog in a single `/atw.embed` run.
- **Credential-source authority.** Every authed catalog entry's `credentialSource` MUST be derived from an OpenAPI `security` declaration (R3 / FR-013). The build halts when the derivation cannot be satisfied.
- **Origin alignment.** `project.md#storefrontOrigins`, the backend's `ALLOWED_ORIGINS` at render time, `demo/shop/backend`'s CORS default, and `host-requirements.md#CORS origins` MUST all resolve to the same value set. The single source is `project.md`.
- **Tool-name resolution.** The `tool_name` carried in a `ToolResultPayload` MUST resolve to an operationId present in the deployed backend's manifest, which in turn MUST match an entry in `action-executors.json`. A `tool_name` that does not resolve is a widget bug; the backend 400-rejects.

## Entities explicitly NOT changed

- `ConversationTurn.content`: remains `string` (spec Assumptions + FEATURE-008 Non-goals). No migration to `string | ContentBlock[]`.
- `atw_postgres` schema: unchanged — no new tables, no new columns.
- `shop_postgres` schema: unchanged — this feature only patches `demo/shop/backend/src/index.ts` for CORS; no Prisma schema changes.
- Feature 003 chat-endpoint response shapes (`text`/`citations`/`pending_turn_id` and `action_intent`/`pending_turn_id`/`tool_call_budget_remaining`) are preserved; the new `response_generation_failed` shape is additive.
